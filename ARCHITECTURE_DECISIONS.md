# Architecture Decisions

A running log of significant architecture/requirements decisions for BTSBootcamp, in the order they were made, with the reasoning behind them. Not a spec — the spec is `BTSBootcamp-Requirements.md` and `CLAUDE.md`. This is the "why," so future contributors don't have to re-derive it or accidentally re-litigate something already settled.

Each entry: what was decided, what alternatives were considered and rejected, and why.

---

## 2026-07-18/19 — User writes: three-repo credential-isolation pipeline

**⚠️ Superseded 2026-07-21 — see the entry below ("Google-Form-intake write pipeline").** The client-embedded PAT this design relied on doesn't actually stay alive: GitHub auto-revokes its own PAT-format tokens the moment they're detected in a public repo, regardless of scope, confirmed independently for both this repo's signup token and the later comments token. Kept below for the reasoning that's still relevant (why a *single* repo-wide token was rejected, the latency trade-off) — just know the specific mechanism (client-embedded PAT → `burnthestage`/`campcomments`) is dead and replaced.

**Decision:** User signups (and likely comments, when built) don't write to the main site repo at all. Instead:
- `btsbootcamp` (this repo) — site code only. No write credential used anywhere in the pipeline ever has permission on this repo.
- `burnthestage` — staging repo. The public-facing write goes here: a fine-grained GitHub PAT scoped *only* to this repo (safe to embed in client JS, see `js/auth.js`) creates a new file per signup under `pending/`.
- `bestofbootcamp` — the real, live data repo. A GitHub Actions workflow in `burnthestage` (`.github/workflows/validate-and-promote.yml` + `scripts/promote.js`) validates pending submissions (valid JSON, matches schema, no duplicate username) and, using a *second*, separate credential (`BOB_TOKEN`, a repo secret scoped only to `bestofbootcamp`), promotes valid ones into `bestofbootcamp/data/users.json`. Invalid/rejected submissions are deleted from staging without being promoted.

**Why two credentials instead of one:** GitHub fine-grained PATs scope to a whole repo, not a branch — a single token with access to a repo containing both a "pending" and "live" area could still be misused to bypass any intended workflow, and enforcing separation via branch protection requires configuration that's easy to get wrong. Two separate repos make the boundary enforced by GitHub's permission system itself, with nothing to misconfigure.

**What this protects against:** code defacement / script injection (impossible — no write credential in this pipeline touches the code repo) and structural data corruption (invalid JSON, wrong schema — caught before promotion). What it does NOT protect against: semantically-valid-but-garbage content (a well-formed spam row) — that needs smarter heuristics (e.g. rate-limiting new rows), not just the repo boundary.

**Accepted trade-off — latency:** writes are not instant. Realistic end-to-end time is roughly 30 seconds to a couple of minutes (Actions trigger/queue delay + validation run + promotion commit + propagation to wherever the live site reads from). Accepted explicitly in exchange for the security isolation — see "why this is acceptable" below.

**Why the latency is acceptable:** accounts/comments are an optional social layer, not required for core site use — browsing, watching, and the Bootcamp path all work with zero login. The friction of an async signup doesn't sit in the onboarding funnel.

**Alternatives considered and rejected:**
- *Single client-embedded GitHub PAT with repo-wide access* — rejected: a leaked token could rewrite any file in the repo (fine-grained PAT "Contents" permission isn't scoped per-file), including the site's actual code.
- *IP-address-based user tracking/identity* — rejected: IPs are unstable (rotate on mobile/dynamic connections) and shared (NAT/household/carrier-grade NAT collapses multiple people into one identity); also carries real privacy weight if published in a public repo's history, more so than a self-chosen username.
- *Progressive "security questions" unlocked by Bootcamp progress* (gamified auth) — logged as a future idea on GitHub issue #7, not adopted for MVP. Revisit once the Bootcamp chapter structure (#6) exists to unlock attributes against.
- *Pure human-moderated email/mailbox queue* — considered viable (friction judged acceptable), but the multi-repo pipeline was judged better: no manual-labor bottleneck, comparable security properties once the token-exposure risk was fully broken down, and it doesn't depend on an external email provider tolerating automated IMAP access.
- *A Vercel serverless function (`api/create-user.js`) holding a single repo-wide secret* — this was the original plan and was built, then removed once the repo-isolation design made a client-embedded (but narrowly-scoped) token an acceptable replacement. See "hosting" decision below for why Vercel was dropped entirely.

## 2026-07-19 — Personal data (progress tracking) is local-first, not written anywhere

**Decision:** Watch progress, favorites, and PIN checks are stored only in `localStorage`, per browser/device. No write pipeline, no repo, no server, ever, for this data.

**Why:** this data only needs to matter to the person generating it — there's no requirement for other people or other devices to see it. Given that, the entire write-credential/security conversation above doesn't apply, because nothing needs to be shared.

**Trade-off accepted:** progress does not sync across devices or browsers, and clearing site data loses it. This is a real departure from the original PRD (which put progress in `users.json` for cross-device durability via the profile system) but was judged the better fit given the "genuinely static, old-school" direction.

**Consequence for the roadmap:** issue #13 (user-based recommendation signals) assumed visibility into individual watch history that this decision doesn't provide. That issue's scope needs revisiting — it would have to work off aggregate/anonymous signals, or the local-first decision would need a carve-out, if it's ever built.

## 2026-07-19 — Hosting: GitHub Pages, not Vercel

**Decision:** Site hosting moves to GitHub Pages. Vercel (and the serverless function it was hosting) is no longer part of the architecture.

**Why:** the original reason for Vercel was `api/create-user.js` needing a host that supports serverless functions — GitHub Pages doesn't. Once the three-repo write pipeline removed the need for that function entirely (see above), there was nothing left requiring Vercel. GitHub Pages is also the better philosophical fit for "genuinely static, old-school" hosting, and it keeps everything (Pages + Actions + all three repos) under one vendor, which isn't a hard requirement but is a nice property to get for free.

**Consequence:** all repos are kept **public** — this doesn't weaken the credential-isolation design at all (isolation comes from permission scope, not repo visibility) and unlocks unlimited free GitHub Actions minutes (private repos get a capped free allowance), on top of matching the project's existing "no privacy guarantees, radically transparent" ethos already established for user data.

**Consequence for docs:** `CLAUDE.md`'s hosting section and file structure needed updating to drop Vercel references; the local `data/users.json` in this repo is removed since it's no longer the live data source (that's `bestofbootcamp/data/users.json` now).

## 2026-07-19 — Cross-category tagging and song-linking are computed from title text, not hand-maintained

**Decision:** `scripts/build_videos_json.py` now derives two things automatically from each video's title, replacing what used to be manual/nonexistent:
- `tags` — every category a video structurally matches, beyond its source-playlist `type`, with **no category treated as primary**. A video's `type` is still one concrete fact (which playlist it actually shipped in), but `tags` can hold any number of additional categories with equal standing.
- `song` — the quoted title substring (e.g. "Butter"), used purely as a recommendation-linking key across categories, not a classification.

**Why title text, and why NOT a loose keyword list:** matching was validated against the real CSVs before being built, not assumed. Plain substring matching on category names over-matches badly — e.g. "MV" appears in 25 Bangtan Bomb titles that are about an MV ("MV Reaction," "MV Shooting") but aren't MVs themselves, and "dance practice" (any case, anywhere) matches casual bomb titles like "Attack on BTS at dance practice" that aren't Dance Practice videos. The fix: each category's pattern is derived from the **exact naming convention that category's own official videos use** (e.g. real MVs are titled `'[Song]' Official MV`; real Dance Practice videos are titled `'[Song]' Dance Practice`), and a video only gets cross-tagged if it matches that specific convention, not a loose mention. This produces exactly the previously-known 5 Bangtan-Bombs-that-are-actually-Dance-Practice videos with zero false positives, and correctly produces zero MV cross-tags (no bomb follows the real MV convention, confirming playlist origin was already correct there — not a broken rule, an expected negative result).

**Why `song` is separate from `tags`:** tags answer "what type of content is this," song answers "what release is this about," and they don't collapse into each other — an MV, its Dance Practice, and a Bangtan Bomb about its jacket shoot all share the same `song` while having three completely different `type`s. Verified against real data: "Butter" alone links 16 videos across all 4 non-Run-BTS playlists; 115 songs link 2+ videos catalog-wide. This became a new, high-priority carousel in the player's recommendation river (`More '[Song]'`), placed ahead of "Up Next" — sharing a release is a stronger signal than sharing an era or type.

**Known limitations, accepted rather than solved:**
- Only Dance Practice and MV have validated cross-tag patterns so far. Behind/Sketch/Episode-style categories likely have the same casual-vs-formal ambiguity (e.g. "Sketch" appears in both Bangtan Bombs and BTS Episodes without being a reliable distinguishing signal) and weren't given patterns — better to leave a category untagged than guess at a pattern that hasn't been checked against real titles.
- `song` extraction can truncate early on a song title containing its own apostrophe (e.g. "Killin' It Girl") since the pattern uses quote characters as delimiters. Accepted: a missed link just means one fewer recommendation, not incorrect data.
- Song-name casing was verified consistent across the real catalog (grouping coverage was identical with/without lowercasing), so `song` keeps original case for cleaner display rather than normalizing defensively against a problem that doesn't occur in practice.

## 2026-07-21 — Comments (#15): a peer staging repo, not routed through burnthestage; profile-required; V1 scope cut to flat per-video comments

**Decision:** Comments extend the existing write-isolation pattern with a fourth repo, `campcomments`, rather than reusing `burnthestage`. `campcomments` is a **peer staging inbox** to `burnthestage` — the two are independent, neither routes through the other — and both promote into the same destination, `bestofbootcamp`, which now holds both `data/users.json` and `data/comments.json`.

**Why a new staging repo instead of adding a `pending-comments/` folder to `burnthestage`:** two reasons, both about risk shape, not raw security:
1. **Blast radius for code changes, not just credentials.** `burnthestage`'s signup pipeline was just verified live end-to-end this session (#16). Adding comment-promotion logic to that same repo — new script, new workflow file — means any future bug in comment handling shares a repo with the already-proven signup path. A separate repo means comments can never affect signups, full stop, regardless of what the code does.
2. **The security shape is already proven, just reused.** `campcomments` mirrors `burnthestage` exactly: a client-embedded token scoped only to itself, holding nothing but an unvalidated `pending/` queue. This isn't a new risk category being introduced — it's the same accepted risk (a leaked client token can spam a worthless inbox) applied to a second inbox, rather than inventing something novel.

**Why `bestofbootcamp` stays the single, consolidated destination rather than also splitting per data-type (e.g. a `campcomments`-owned `data/comments.json`):** the isolation boundary in this architecture was never "one live-data repo per content type" — it's "code repo vs. staging repo(s) vs. validated-data repo." `BOB_TOKEN` already has full `contents:write` on all of `bestofbootcamp` (fine-grained PATs scope per-repo, not per-file), so adding `comments.json` there doesn't expand what that credential can already do. A separate live-data repo for comments would mean a *fifth* repo and a *third* server-only credential for no corresponding security gain. Each staging repo issues its own `BOB_TOKEN` (same name, same scope, different token value per repo, since GitHub secrets aren't shareable across repos) — comment-promotion also uses this to cross-check `username` against the live `users.json` in the same repo, rejecting comments attributed to a username with no real profile.

**Why posting requires an existing profile:** matches the PRD as written (`BTSBootcamp-Requirements.md` §11: "Only users with a profile can post — comments attributed to username"). An anonymous, free-text-name alternative was floated mid-session but rejected: this repo has **no comment moderation system built** (PRD explicitly lists it as an open question), and with a public write path and zero moderation, anonymous posting is a much easier spam/impersonation target than profile-gated posting, where at least a username ties abuse to a revocable account. The PRD's nested-reply feature also needs persistent identity to mean anything — anonymous posters can't meaningfully be replied to across sessions.

**Consequence for UX — draft persistence across the login redirect:** since posting requires login, a logged-out visitor who types a comment and hits Enter is redirected to `profile.html` rather than shown an inline error. To avoid losing what they typed, the draft is saved to `localStorage` (`bts_pending_comment_draft`, see `js/comments.js`) and auto-posted the moment login succeeds — not on signup submission itself, since a brand-new account still has to clear the async staging/promote pipeline before it's real.

**V1 scope, deliberately cut down from the full PRD spec:** flat, per-video comments only — no `interval` field (no 10-second-interval bucketing), no nested replies (schema keeps `parent_comment_id` for forward-compatibility but the promote script currently rejects any non-null value), no floating "top comment" bubble overlay, no full-timeline scrubbing, no local per-comment likes. Chosen to get the core write path (staging → validate → promote → read-back) proven live first, the same way user profiles shipped as a working pipeline (#7) before being hardened (#16) — rather than building the full interval-threading UI against an unproven write path. These become explicit, scoped follow-up work once V1 is verified live.

**Discovered during live verification — a second, CDN-level delay beyond promotion, and a local-echo fix:** the existing "writes take ~30s-2min" framing (see the three-repo pipeline decision above) only accounts for the Actions promotion step. Live testing of comments surfaced a second, independent delay: `raw.githubusercontent.com` (used for every read in this architecture — `loadUsers()`, `loadComments()`) caches per-CDN-edge for up to 5 minutes (`cache-control: max-age=300`), and different requests — even from the same machine seconds apart — can land on different edges with independently-stale cache state. A `curl` from one location returned the freshly-promoted comment immediately while the browser, hitting a different edge, kept returning stale (pre-comment) data for several minutes after promotion had already succeeded. This almost certainly already applied to `users.json` reads too; it simply went unnoticed there because nothing had previously reloaded the exact same URL within the same browser tab both before and immediately after a promotion.

**Decision:** rather than fight CDN propagation (not something this architecture controls), comments this browser itself just posted are echoed locally — `js/comments.js`'s `saveLocalComment()`/`pendingLocalComments()`, `localStorage` key `bts_local_pending_comments`, 10-minute TTL — and merged into the display ahead of the live fetch until the live data actually contains them (matched by video_id + username + comment text, since the client never learns the server-assigned `comment_id`). This makes "you see your own comment immediately" independent of both the promotion delay and CDN propagation, without pretending to solve either — other people still only see a new comment once both delays clear, which remains an accepted trade-off per the original latency reasoning.

## 2026-07-21 — Google-Form-intake write pipeline (supersedes the three-repo credential-isolation pipeline)

**What broke:** both `STAGING_TOKEN` (`js/auth.js`) and `COMMENTS_TOKEN` (`js/comments.js`) went dead (`401 Bad credentials`) shortly after each was pushed to this public repo — confirmed via direct API calls, confirmed independently for both tokens on two separate occasions. Root cause: GitHub auto-revokes its own PAT-format tokens (`github_pat_...`) whenever detected in a public repo's content, regardless of scope or how the repo's push-protection/secret-scanning-alert settings are configured (confirmed: this repo has secret-scanning *alerts* off, revocation happened anyway — it's a token-issuer-side action, not a repo setting). Obfuscating the token client-side (base64 or a custom cipher) doesn't fix this: the decode logic has to ship in the same public file, so anyone can reverse it in seconds regardless of whether GitHub's scanner catches it — the real vulnerability was never "GitHub can read it," it's "anyone can," which no encoding changes.

**Full investigation, live in GitHub issue #18** (`diyamaxxing/btsbootcamp#18`): every alternative considered and why each was closed — a serverless/edge proxy (ruled out, conflicts with this project's no-backend stance), GitHub Issues as the submission mechanism (ruled out, requires visitors to have a GitHub account), the token living only on the architect's own device (closed — either reintroduces the already-rejected human-moderated-queue bottleneck, or requires self-hosting always-on infrastructure, which is worse than a managed option, not better), a GitHub Gist as an anonymous inbox (closed — authenticated gist creation has the identical credential problem, anonymous gist creation isn't discoverable afterward). Also mapped there: GitHub Actions' full trigger surface — `schedule` (cron) is the *only* trigger type needing zero credential from an anonymous caller, but a scheduled job still needs something to hold a submission until the next run, which is the gap every rejected idea ran into.

**Decision:** a Google Form is the "something." Visitors submit straight to it — configured for anonymous responses, no Google account needed, no credential of any kind ships to the browser. `js/auth.js`/`js/comments.js`'s `submitToGoogleForm()` does a real hidden-`<iframe>` `<form>` POST (not `fetch`) directly to the form's own `.../formResponse` endpoint — a form POST sidesteps Google's lack of CORS headers on that endpoint, which a `fetch()`/XHR read would be blocked by. Every response becomes a row in the form's linked Google Sheet. A `schedule`-triggered (every 5 min) + `workflow_dispatch`-enabled GitHub Actions workflow in `bestofbootcamp` reads new rows via the Sheets API v4, using a Google service-account credential that lives *only* as a repo secret (`GOOGLE_SERVICE_ACCOUNT_KEY`) — never client-exposed, so nothing for any scanner to ever catch. Auth is hand-rolled (JWT signed with Node's built-in `crypto`, no `googleapis`/`google-auth-library` dependency — see `automation/lib/google-sheets.js`) to keep this project's zero-npm-dependency convention intact.

**Consequence — repo consolidation.** The entire reason `burnthestage`/`campcomments` existed as *separate* staging repos was to contain the blast radius of a client-embedded write credential: one repo per credential, so a leak could only ever touch one narrow thing. There is no client-embedded credential anymore, so that isolation requirement is gone. Everything server-side now lives in **`bestofbootcamp`** alone — `automation/signups/promote.js` and `automation/comments/promote.js`, organized by folder rather than by repo (mirrors the two Sheets/Forms), each with its own workflow file since GitHub Actions only scans `.github/workflows/` directly, not subfolders. Both scripts write straight to `bestofbootcamp`'s own `data/*.json` on disk (already checked out by the workflow) and `git commit`/`push` using the workflow's auto-provided `GITHUB_TOKEN` — no more cross-repo Contents-API PUT/GET-sha dance, since destination and automation now live in the same repo. `burnthestage`/`campcomments` are unused as of this change; archiving/deleting them is a separate, later decision, not automatic.

**Accepted trade-offs:**
- **Polling floor, not instant.** The old `push`-triggered workflow ran the moment a file landed; `schedule` can't go faster than ~5 minutes. Submissions now wait up to that long before even being *looked at*, on top of whatever the promotion run itself takes. The local-echo mechanism in `js/comments.js` (see the entry above) matters more now, not less — it's the only thing that makes a person's own submission feel immediate.
- **A "Processed" column, not a deleted file.** The old pipeline deleted pending files after handling them (accepted or rejected) so nothing got reprocessed. Sheets don't have an equivalent "delete row via a script running elsewhere" primitive that plays nicely with Forms still appending to the same sheet, so `automation/lib/google-sheets.js` marks a `Processed` column instead (auto-added on first run) — same "never retry a handled submission" property, different mechanism.
- **Entry IDs aren't a documented, stable API.** Google's `.../formResponse` endpoint and each question's `entry.XXXXXXXXX` field name are the same thing the *real* Google Form page itself posts to under the hood — extremely widely used, stable in practice for years — but not a versioned, Google-supported public API. If a form's questions are ever edited (added/removed/reordered), the entry IDs need re-extracting and `SIGNUP_FORM_FIELDS`/`COMMENT_FORM_FIELDS` in the client code need updating to match.

## 2026-07-21 — Framework migration: vanilla HTML/CSS/JS → Next.js (static export), superseding "No framework (for now)"

**⚠️ Amended the same day, once the migration was actually executed — see the entry below ("Framework migration executed").** Two things below turned out not to match reality once the real code was read closely (not assumed from documentation): the recommendation river is **9 carousels, not 10** (the code's own comment already flagged the doc as stale — CLAUDE.md's routes table had the right number, its prose didn't), and the **build-time-data-read plan was reversed** — content data (`videos.json`/`eras.json`) stayed a runtime `fetch()`, not a build-time bake-in, for reasons explained in the execution entry. Kept below for the reasoning that's still valid (why Next.js specifically, why static export, why not reopen the hosting decision).

**Decision:** the original "no framework" call (see `CLAUDE.md`) was explicitly a *staging* decision, not a permanent one — "feel the architecture and data flow before introducing a framework." That condition is now satisfied: the write pipeline, recommendation river, browse filters, auth, and comments have all been built and verified live end-to-end against the flat-JSON/GitHub-as-database architecture. The architect is calling that phase done and moving to Next.js (React). This entry records the decision; page-by-page migration order is separate follow-up planning, not decided here.

**Why now, specifically:** two concrete blockers, not a desire to use a framework for its own sake:
- **UI can't grow further on the current base without real duplication cost.** `scoreVideo()` is hand-copied into every page that needs it (see `CLAUDE.md` Conventions), and the player's 10-carousel recommendation river is 10 near-identical hand-written blocks in `player.html` rather than one component parameterized 10 ways. Both were acceptable while the site was small; both get worse, not better, with every new page.
- **The Bootcamp path (#6) — the last major unbuilt feature — is a guided, stateful, multi-step flow.** That shape is exactly what a component/route model is for, and exactly what's painful to hand-roll in stubbed vanilla HTML. Building it on the current base would mean either duplicating even more hand-rolled state logic or building it once and immediately wanting to migrate — better to migrate first.

**Why Next.js specifically, not Vue/Svelte/Astro:** this wasn't a fresh evaluation — `CLAUDE.md`'s original "no framework" entry already committed to a target: "CSS and JS modules are organized to migrate cleanly to React/Next.js later" and "`js/` files map 1:1 to future React hooks/utilities." Picking a different framework now would throw away that alignment for no benefit; Next.js was the planned destination since the vanilla-JS phase began.

**Consequence — the hosting decision constrains the migration shape.** GitHub Pages over Vercel was itself a deliberate, already-settled call (see the 2026-07-19 entry above) made for "genuinely static, old-school" hosting and single-vendor simplicity — this migration does not reopen that. Concretely, that means Next.js runs in **static export** mode (`output: 'export'`), not full SSR:
- No server components doing runtime data fetching, no API routes, no ISR/on-demand revalidation — every route still has to emit as a plain static HTML file that GitHub Pages can serve as-is.
- `data/videos.json`/`eras.json` (this repo, content data) move from a runtime `fetch()` to a **build-time** read, baked into the static output — conceptually the same moment `scripts/build_videos_json.py` already runs today, just consumed by `next build` instead of committed as a fetchable file.
- `bestofbootcamp`'s `data/users.json`/`data/comments.json` (user-generated data) **stay runtime, client-side fetches** against the GitHub raw-content URL, unchanged — that data changes without a site rebuild, so it can't be baked in at build time the way content data can.
- A build step now exists where the deploy previously had none: a GitHub Actions workflow needs to run `next build` (static export) before Pages serves the result. Actions is already free/unlimited on this repo (public repo, per the hosting decision), so this is one more workflow, not a new cost or constraint.

**What does NOT change — this is a presentation-layer migration, not an architecture rewrite:** the Google-Form-intake write pipeline, the `bestofbootcamp` data split, local-first `localStorage` progress tracking, and the flat-JSON-as-database model are all framework-agnostic and untouched by this decision. Nothing about *how data is written or where it lives* is being revisited here.

**What migrates near 1:1 (this is what the original "map 1:1 to future React hooks" framing bought):**
- `js/auth.js`, `js/comments.js`, `js/progress.js` → `useAuth`, `useComments`, `useProgress` hooks. The Google-Form POST, `localStorage` session pointer, and comment local-echo mechanism (`saveLocalComment`/`pendingLocalComments`) port as-is; only where the logic lives changes.
- The per-page-duplicated `scoreVideo()` collapses into one shared function, imported everywhere instead of copy-pasted.
- The 10-carousel recommendation river becomes one parameterized `<Carousel>` component instead of 10 hand-written blocks.
- Browse filter state (`?types=`, `?members=`, `?eraFrom=`/`?eraTo=`, `?yearFrom=`/`?yearTo=`) maps onto Next.js's `useSearchParams`/`router.push` instead of hand-rolled URL parsing.

**Alternatives considered and rejected:**
- *Vue, Svelte/SvelteKit, Astro* — all viable frameworks in the abstract, but rejected because the codebase was already organized against a specific target (React/Next.js) since the "no framework" decision was made; switching targets now would discard that alignment for no corresponding gain.
- *Staying on vanilla HTML/JS and hand-rolling more structure (a lightweight templating layer, manual component-like patterns)* — rejected: this defers the same eventual migration while adding more hand-rolled surface area (more copy-pasted carousels, more manual URL parsing) that would just have to be unwound later.
- *Migrating hosting off GitHub Pages (e.g. back to Vercel) to unlock full Next.js SSR* — not adopted here; the 2026-07-19 hosting decision is treated as still settled. Static export keeps the migration scoped to presentation-layer only. Revisit only if a concrete feature need arises that static export genuinely can't support.

## 2026-07-21 — Framework migration executed: TypeScript + Tailwind v4 + Next.js App Router, static export verified locally

**What happened:** the decision above was carried out in full, in one pass, in the same session it was made — every page and `js/` module ported, old vanilla files deleted, `npm run build` producing a working static export. Two real corrections surfaced only by reading the actual code and running the actual build, not by trusting the existing docs or planning in the abstract — both are why "verify, don't assume" mattered here as much as the port itself.

**Reversed: content data stays a runtime fetch, not a build-time bake-in.** The decision entry above planned to move `videos.json`/`eras.json` to a build-time read. Once the real pages were read closely, this stopped making sense: every interactive surface (browse's filter grid, the player's recommendation pooling) needs the *entire* ~2,800-video catalog in the browser regardless of framework — that was never going away. Embedding that JSON into a Server Component's payload would mean either bloating every route's initial HTML/RSC payload with the same ~1MB of data, or engineering shared-chunk deduplication across routes to avoid it — real complexity, for a benefit (avoiding one client-side fetch) that the current design already gets from something free: the browser's ordinary HTTP cache on a stable `/data/videos.json` URL, which a build-time-embedded payload would forfeit. So `public/data/videos.json`/`eras.json` (relocated from `data/` — see "why `public/`" below) stay exactly what they were: a plain static asset, `fetch()`'d client-side once per session and cached by the browser like any other static file. `bestofbootcamp`'s `data/users.json`/`data/comments.json` were never a candidate for build-time embedding either way (they change without a site rebuild) and are unaffected.

**Why `public/data/`, not `data/`:** Next's static export only serves files placed under `public/` (or emitted by the build) at their own path — a plain `data/videos.json` at the repo root is not automatically part of `out/`. Rather than add a copy-into-`public/` step to the build (one more thing to keep in sync, one more way for a stale copy to ship), `scripts/build_videos_json.py`'s output path moved to `public/data/videos.json` directly, so there's a single source-of-truth location for the file the site actually serves. `data/raws/*.csv` (the source CSVs) stayed under `data/` — only the generated/served files moved.

**Corrected: the recommendation river is 9 carousels, not 10.** Read `pages/player.html` directly rather than trusting either the decision entry above or CLAUDE.md's own prose sections (both said 10): `buildRecommendations()`'s own code comment already flagged that the "5/6 fan favorites" numbering only ever built one carousel, and CLAUDE.md's routes table already had the correct count (9) — only the spec section and changelog were stale. Fixed in the ported `lib/recommendations.ts`, and in CLAUDE.md's remaining stale mentions, so there's one canonical number going forward.

**Other things found only by reading the real code (not by extrapolating from docs):**
- The recommendation river was *already* well-factored in the vanilla version (`buildRecommendations()` + one shared `pool()` helper + one `renderCarousel()` template) — not 10 hand-duplicated blocks as the original decision entry assumed. This made the port to `lib/recommendations.ts` + one `<Carousel>` component more of a straight lift than a rewrite.

## 2026-07-22 — Import from the external "BTS Content Index" sheet: read-only, staged, and classified as tags rather than new types

**Decision:** a fan-maintained, externally-owned Google Sheet ("BTS Content Index," link-viewable, not editable by this project) turned out to hold ~15k rows of archival BTS content, most of it never captured by the six known YouTube playlists `fetch_playlists.py` pulls from. After scoping, diffing, and metadata-backfilling the real net-new videos, 5,869 of them were imported into `data/raws/content_index.csv` as a seventh source feeding `build_videos_json.py`, growing the catalog from 2,767 to 8,636 videos. This is a one-time (so far) import, not a live dependency — the site never talks to this sheet or any Google API at runtime; everything happens in these build-time scripts, same as every other raw CSV.

**Why a multi-stage pipeline instead of one script:** each stage's output was worth inspecting before trusting the next one with real API calls or a merge into live data:
1. `scripts/inspect_content_index_sheet.py` — row/column-layout scoping via Google's public **gviz** JSON endpoint, which needs zero credentials for a link-viewable sheet. This is what caught that only 19 of ~24 tabs share one consistent `Date / Title / YouTube / Weverse / Other` column layout — `Weverse Live`, `Music Shows`, `Series Compilation`, and `Performances` each use a different layout and were explicitly left out rather than guessed at.
2. `scripts/diff_content_index_sheet.py` — real hyperlink extraction via **Sheets API v4** (gviz only exposes visible cell text, e.g. "HERE," never the URL behind it), diffed against every video ID already in `data/raws/*.csv`. This is also where a real bug was caught and fixed: A1-notation ranges for a sheet name containing special characters (e.g. `LYS:H`, `LYS:T`) need explicit single-quoting, or the API either silently returns 0 rows or 400s — verified against all 19 tabs individually rather than trusting an unquoted range to "probably work."
3. `scripts/fetch_content_index_metadata.py` — batches the net-new IDs (6,523 found) through the YouTube Data API for real `upload_date`/`duration_sec`/`view_count`/`like_count`/status, same `videos.list` batching pattern `fetch_playlists.py` already uses. Writes to a **staging** file, not `data/raws/` — 654 of the 6,523 IDs (10%) came back empty (deleted or private-and-inaccessible), which needed to be known before anything got merged.
4. `scripts/build_content_index_csv.py` — only after the staged metadata existed did the actual `data/raws/content_index.csv` get generated, deduplicated by YouTube ID (the sheet sometimes lists the same video under two era tabs).

**Why the read access needed an API-key change, and why that's still fine to keep read-only:** `YOUTUBE_API_KEY` (the existing key `fetch_playlists.py` uses) was originally restricted to YouTube Data API v3 only, and Sheets API v4 calls 403'd with `API_KEY_SERVICE_BLOCKED`. The architect enabled Google Sheets API for the same Google Cloud project and added it to that key's API restrictions — no new credential, no elevated scope: an API key is a per-project quota/attribution identifier, not a capability grant, and every call this pipeline makes is a read (`spreadsheets.get`) against a sheet the project doesn't own and can't write to regardless of what the key allows.

**Why `type="Archive"` for every imported video, with real classification pushed entirely into `tags`:** this batch is a heterogeneous mix (pre-debut solo tracks, brand interviews, broadcast cuts, member vlogs) with no single clean source playlist — exactly the situation `BTS On Air` was already in (see the cross-tagging decision above), which was resolved the same way: one `type` for provenance, `tags` for the actual content classification, checked with equal weight against every video regardless of source. This was a deliberate call, not a default — the alternative (a new `type` per detected category, e.g. `type="Interview"`) was considered and rejected because `type` and `tags` already get equal treatment in every filter/carousel surface (`pages/index.html`'s type-dropdown and `mainmuster.html`'s carousels both build their list from `type` **and** `tags` combined — see the video-schema tags note in CLAUDE.md), so there's no browsability lost by choosing tags, and it keeps the "one `type` = one concrete content source" invariant intact for every existing category.

**New `CATEGORY_PATTERNS` entries, validated against the real fetched titles before landing (same discipline as the On Air additions, not guessed):** `Interview`, `Behind the Scenes`, `Log`, `Teaser`, `Trailer`, `Preview`, `Countdown`. Two things surfaced only by checking real titles:
- `Countdown`'s dominant false-positive shape was the *"M Countdown"/"M!Countdown"* broadcast show name (already covered by the existing `Music Show` pattern), not genuine pre-release countdown content — the pattern excludes it with a fixed-width negative lookbehind, `(?<!m[ !-])\bcountdown\b` (Python's `re` module requires lookbehinds to be fixed-width; a first attempt using a `{0,2}`-width character class failed to compile).
- `Log` (BTS's own member-vlog series — `"{Member}'s Log"`, `[BE-Log]`, `[V VLOG] V-log`) turned out to need the loose `\blog\b` rather than a tighter possessive-form pattern: tightening it to require `'s Log` specifically lost 14 genuine matches that use `BTS' Log` (apostrophe-only, no "s") or the `[BE-Log]`/`V-log` bracket conventions instead — checked directly against the real title set rather than assumed, exactly the same "verify before landing" bar the Dance Practice/MV patterns were held to.
- `Log` ended up the single largest new tag at 198 matches, ahead of `Interview` (193) and `Behind the Scenes` (192).
- A `Message` category was considered and dropped — real title samples were too heterogeneous (mixing show segment names, brand-column titles, and genuine greetings) to trust as a distinguishing pattern, following the same "leave a category untagged rather than guess" rule the cross-tagging decision already established for `Sketch`/`Episode`.

**Known limitation, found but not fixed here — since fixed, see below:** merging in `content_index.csv` surfaced 40 pre-existing duplicate videos (same YouTube ID, two different catalog rows) among the *original* six CSVs — e.g. a Bangtan Bomb also present in the Dance Practice playlist. Confirmed none of the 40 touch the new `Archive` import. Left alone in this entry since it predates this change; collapsed in a same-day follow-up, see immediately below.

## 2026-07-22 — Collapse duplicate videos into one entry, and make it self-maintaining for future pulls

**Decision:** the 40 duplicates found above split into two genuinely different causes, so they got two different fixes rather than one blanket "delete the second row":

- **32 same-type duplicates** — the exact same video listed *twice within one official YouTube playlist* (identical title/type/view/like counts, adjacent playlist positions). Zero information loss either way: `scripts/collapse_duplicate_videos.py` just drops the later row.
- **8 cross-type duplicates** — the exact same video legitimately listed under *two different* official playlists (e.g. a Bangtan Bomb also in the Dance Practice playlist), with no textual marker in the title for `build_videos_json.py`'s pattern-based tagging to recover on its own. Dropping either row outright would silently lose real category information. Instead: the row from whichever CSV comes first in `CSV_FILES` processing order is kept as canonical `type`, and the other type is written into a new `extra_tags` CSV column on that row — merged into the computed `tags` array at build time (`compute_tags()` now takes an optional `extra_tags_raw` param). This is the one place `tags` isn't purely title-derived, and it's deliberately narrow: `extra_tags` exists only to record "this exact video was also officially filed under playlist X," a fact no title pattern could ever recover, not a general hand-tagging escape hatch.

**Why collapse instead of leaving both rows filterable-but-separate:** two catalog rows for one video means it can appear twice in browse results and twice in the same recommendation carousel — worse UX than the duplicate-listing fact being invisible. Preserving the second type as a tag (rather than just deleting it) keeps that fact discoverable without the UX cost.

**Made self-maintaining, per architect request, rather than a one-time manual fix:**
- `fetch_playlists.py`'s `fetch_all_playlist_items()` now dedupes by `video_id` within a single playlist fetch, so a live YouTube playlist that lists the same video twice (the 32-group cause) can't reintroduce those duplicates on a future refetch.
- `fetch_playlists.py`'s `main()` now runs `collapse_duplicate_videos.py` automatically as its last step. This catches the cross-playlist case (the 8-group cause) too, which per-playlist dedup structurally can't see — each playlist is still fetched independently, so the same video landing in two different output CSVs is only detectable after all of them exist on disk.
- `fetch_playlists.py`'s existing "preserve manually-filled fields from existing CSV" logic (already used for `era`/`members`/`subtitles`/`description`) now also preserves `extra_tags` across a refetch, so re-running the fetcher doesn't wipe a previously-recorded cross-playlist tag.

**Result:** catalog goes from 8,636 to 8,596 videos (40 fewer rows, zero fewer real videos), 0 duplicate video-id groups remaining — verified directly against `public/data/videos.json` post-rebuild, not assumed from the script's own success message.
